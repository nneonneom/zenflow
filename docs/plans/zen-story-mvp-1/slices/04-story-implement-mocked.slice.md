### `story-implement` Behavior

1. Source State Store; read `current_slice` and `total_slices` from `state.json`
2. For each slice from `current_slice` to `total_slices`:
   - Load slice file from `~/.zenflow/$story_id/slices/`
   - Present slice goal and tasks
   - Implement the code changes described in the slice
   - If input is needed mid-slice: send Teams notification and wait for user reply
   - On completion: update `status.md` (mark slice `✅ Complete`), increment `current_slice` in `state.json`
3. When all slices done: send Teams completion notification, return context to `zen-story`

---

### "Mocked" Scope

- **Planning Core is mocked** — the plan being executed was generated by `mock-planning-core.sh` (generic slices). Claude executes whatever tasks are listed.
- **Teams is mocked** — `mock-teams-notifier.sh` prints to stdout instead of sending a webhook. Call sites are identical to what the real adapter will use.
- Implementation itself is not mocked — Claude does real work on slice tasks.

### Error Cases

| Error | Handling |
|---|---|
| Slice file not found | Exit with message listing expected filename pattern |
| `state_write` fails mid-loop | Exit; state reflects last successfully completed slice — safe to resume |
| User declines to continue | Exit cleanly; `current_slice` already incremented — `zen-resume` picks up at right slice |
